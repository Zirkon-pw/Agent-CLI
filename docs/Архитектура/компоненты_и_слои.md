# Компоненты и слои

## Слои системы

Для этого CLI на Go полезно выделять не четыре, а пять уровней.

### Composition root

Это верхний слой инициализации зависимостей.

Он отвечает за:

- загрузку конфигурации запуска;
- создание stores и adapters;
- сборку application services;
- создание root command.

В Go это обычно живет рядом с `cmd/agentctl/main.go` и `internal/bootstrap`.

### CLI слой

CLI слой отвечает за:

- команды и подкоманды;
- флаги и help;
- преобразование input в request DTO;
- печать таблиц, summary и ошибок.

Здесь должны жить группы:

- `help`
- `init`
- `task`
- `template`
- `clarification`
- `result`
- `guidelines`

### Application слой

Application слой координирует use cases:

- создать задачу;
- дополнить задачу встроенным шаблоном;
- запустить задачу;
- сгенерировать файл уточнения;
- прикрепить уточнение;
- возобновить задачу;
- остановить или убить run;
- показать runtime state.

Этот слой знает порядок действий, но не должен знать детали сериализации, работы с терминалом и конкретного backend-а исполнения.

### Domain слой

Domain слой отвечает за инварианты:

- какие статусы существуют;
- какие переходы допустимы;
- как сочетаются шаблоны;
- когда задача считается ожидающей уточнения;
- когда можно создать новый run;
- как связаны task, run, clarification и review decision.

### Infrastructure слой

Infrastructure слой отвечает за:

- файловое хранилище;
- runtime registry;
- агента-исполнителя;
- event stream;
- логирование;
- validation executor.

Именно здесь живет конкретная работа с `.agentctl/`, `os/exec`, локами и heartbeat.

## Основные компоненты

### Workspace loader

Ищет `.agentctl/`, валидирует базовую структуру проекта и загружает:

- `config.yaml`;
- `agents.yaml`;
- `routing.yaml`;
- built-in template registry;
- шаблоны, guidelines и task files.

### Task service

Центральный application service для:

- создания и обновления задач;
- чтения task spec;
- применения built-in и custom templates;
- изменения статуса;
- запуска task lifecycle.

### Template registry

Отвечает за два источника шаблонов:

- built-in templates CLI;
- пользовательские шаблоны из файлов проекта.

Registry должен уметь:

- перечислять шаблоны;
- возвращать метаданные шаблона;
- определять поведенческий эффект шаблона;
- проверять совместимость шаблонов в одной задаче.

### Clarification service

Отвечает за clarification flow:

- создать `clarification_request_*.yml`;
- сгенерировать editable `clarification_*.yml`;
- прикрепить уточнение к задаче;
- обновить task status;
- включить уточнение в context pack для следующего `task run`.

Важно: работа с вопросами не должна быть отдельной командной веткой `task questions`. Это поведение шаблона и application flow.

### Context pack builder

Собирает входной пакет для исполнения:

- task spec;
- подключенные prompt templates;
- `.yml`-уточнения;
- guidelines;
- `must_read`;
- include files и summaries;
- runtime settings;
- validation rules.

### Prompt builder

Формирует итоговый execution contract для агента.

В целевой модели prompt должен быть коротким:

```text
Приступай к выполнению задачи: .agentctl/tasks/TASK-001.yml
Используй контекст из: .agentctl/context/TASK-001/
Сохраняй артефакты в: .agentctl/runs/TASK-001/RUN-001/
```

То есть CLI формирует файловый контракт, а не копирует все содержимое задачи в один большой текст.

### Runtime manager

Управляет активными run:

- регистрирует process или session;
- обновляет heartbeat;
- поддерживает `ps`, `inspect`, `watch`;
- принимает control-команды `stop` и `kill`;
- умеет восстанавливать состояние после рестарта CLI.

### Validation runner

Запускает validation commands и возвращает `ValidationReport`.

### Result store

Сохраняет артефакты run:

- prompt snapshot;
- template snapshot;
- attached clarifications;
- logs;
- diff;
- validation results;
- metadata;
- review result.

## Где должна жить state machine

State machine не должна быть размазана по CLI handlers.

Ее стоит держать в отдельном доменном модуле, например:

- `internal/core/task/status.go`
- `internal/core/task/transitions.go`
- `internal/core/run/status.go`

Этот модуль должен централизованно отвечать за переходы:

- `draft -> queued`
- `queued -> preparing_context`
- `preparing_context -> running`
- `running -> needs_clarification`
- `needs_clarification -> ready_to_resume`
- `ready_to_resume -> running`
- `running -> pausing`
- `pausing -> paused`
- `running -> stopping`
- `stopping -> stopped`
- `running -> killed`
- `running -> validating`
- `validating -> review`
- `review -> completed`
- `review -> rejected`

## Что полезно описать интерфейсами

Через интерфейсы стоит абстрагировать:

- `TaskStore`
- `RunStore`
- `TemplateRegistry`
- `ClarificationStore`
- `WorkspaceStore`
- `AgentExecutor`
- `RuntimeBackend`
- `ValidationExecutor`
- `EventSink`
- `Clock`

Это даст предсказуемые unit-тесты и позволит менять инфраструктуру без перетаскивания логики по пакетам.
