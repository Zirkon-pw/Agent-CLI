# Обзор архитектуры CLI

## Цель архитектуры

Agent CLI на Go должен быть не thin-wrapper над LLM API, а локальным control plane для инженерных задач.

Это означает три базовых свойства:

- задача существует как воспроизводимая файловая спецификация;
- выполнение существует как управляемый runtime-процесс;
- поведение агента задается не вручную в чате, а через шаблоны и артефакты проекта.

## Основная модель исполнения

Система строится вокруг короткого исполняющего контракта.

`task run` в целевой модели не собирает гигантский текстовый prompt вручную, а подготавливает файловую среду и передает исполнителю короткую команду вроде:

```text
Приступай к выполнению задачи: .agentctl/tasks/TASK-001.yml
Используй контекст из: .agentctl/context/TASK-001/
Сохраняй артефакты в: .agentctl/runs/TASK-001/RUN-001/
```

Из этого следуют важные архитектурные решения:

- источник истины находится в файлах;
- CLI отвечает за подготовку среды, а не за длинный inline prompt;
- шаблоны промптов и уточнения подключаются как файловые зависимости;
- run можно воспроизвести и диагностировать без истории диалога.

## Главные архитектурные принципы

### Файлы важнее истории сообщений

Система должна опираться на:

- `task.yml`;
- подключенные built-in и custom prompt templates;
- `.yml`-уточнения;
- context pack;
- runtime metadata;
- run artifacts.

История чата может быть полезной для аудита, но не должна быть обязательным источником истины.

Это же касается тестового слоя: все автоматические тесты и фикстуры должны быть собраны в отдельной директории `tests/`, а не размазаны по production-пакетам.

### CLI это transport layer

Команды `agentctl` должны:

- принять аргументы и флаги;
- загрузить workspace;
- вызвать application use case;
- напечатать результат.

Они не должны самостоятельно реализовывать state machine задачи, шаблоны поведения или runtime orchestration.

### Поведение агента задается шаблонами

Сценарии вроде:

- сначала спроси, если задача неясна;
- сначала построй план;
- работай строго без предположений;
- только исследуй кодовую базу;
- только сделай review;

должны описываться через built-in templates CLI и подключаться к задаче как часть task spec.

### Уточнения это не отдельный режим чата

Вопросы по задаче должны быть разновидностью общего task flow.

Если встроенный шаблон `clarify_if_needed` находит неясность, система:

- создает `clarification_request_001.yml`;
- переводит задачу в `needs_clarification`;
- ждет `clarification_001.yml`;
- после `attach` возвращает задачу в `ready_to_resume`.

### Инфраструктура должна быть заменяемой

Через интерфейсы должны подменяться:

- файловое хранилище;
- агентный executor;
- runtime backend;
- event sink;
- validation runner.

Это особенно важно для тестов и для замены движка исполнения без переписывания бизнес-логики.

## Bounded contexts внутри CLI

Внутри приложения на Go удобно выделять несколько контекстов.

### Workspace и configuration

Отвечает за поиск `.agentctl/`, загрузку `config.yaml`, `agents.yaml`, `routing.yaml` и реестра встроенных шаблонов.

### Task lifecycle

Отвечает за создание, обновление, запуск, возобновление, остановку, отмену и завершение задач.

### Template behavior

Отвечает за реестр шаблонов, их совместимость, порядок применения и итоговый behavior contract.

### Clarification flow

Отвечает за генерацию `clarification_request_*.yml`, создание editable-файла уточнения, attach и возврат задачи в исполнение.

### Runtime control

Отвечает за `ps`, `inspect`, `watch`, heartbeat, locks, stop, pause, kill и recovery после рестарта процесса.

### Results и review

Отвечает за артефакты запуска, validation-отчеты, diff, reviewer decision и повторный прогон задачи.

## Высокоуровневая схема

На верхнем уровне архитектура выглядит так:

1. Пользователь вызывает `agentctl`.
2. CLI слой парсит args и flags.
3. Bootstrap загружает workspace и wiring зависимостей.
4. Application service выполняет use case.
5. Domain слой проверяет инварианты и переходы статусов.
6. Infrastructure слой работает с `.agentctl/`, executor-ом, событиями и runtime.
7. CLI выводит короткий и читаемый результат.

## Центральные типы в Go

В реализации на Go центральными типами должны быть:

- `Task`
- `TaskStatus`
- `Run`
- `RunStatus`
- `PromptTemplate`
- `ClarificationRequest`
- `Clarification`
- `RuntimeState`
- `ValidationReport`
- `ReviewDecision`

Эти типы должны существовать и как модели кода, и как сериализуемые файловые артефакты.

## Команды, которые должны быть легкими

Без тяжелой оркестрации должны выполняться:

- `help`
- `init`
- `task list`
- `task inspect`
- `task ps`
- `template list`
- `template show`
- `clarification show`

Это read-only или near read-only сценарии.

## Команды, которые обязаны идти через orchestration

Через application orchestration должны проходить:

- `task create`
- `task update`
- `task run`
- `task resume`
- `task rerun`
- `task stop`
- `task kill`
- `task cancel`
- `clarification generate`
- `clarification attach`

Эти команды меняют task state, создают run-артефакты или затрагивают runtime.

## Рекомендуемые технологические опоры в Go

Практичный базовый стек для такой реализации:

- `cobra` или совместимый пакет для дерева CLI-команд;
- `yaml.v3` для `yml`;
- `context` для отмены, дедлайнов и graceful stop;
- `slog` для структурированных логов;
- `os`, `io/fs`, `filepath` для локального хранилища;
- `os/exec` для validation и локальных раннеров;
- `sync`, `errgroup`, `atomic` для runtime coordination.

Это не жесткая привязка, но хорошая и реалистичная база для Go-CLI такого класса.
