# Потоки выполнения команд

## Общий каркас выполнения команды

Практически любая команда `agentctl` должна проходить один и тот же шаблон:

1. Parse args and flags.
2. Load workspace and configuration.
3. Build request DTO.
4. Execute application use case.
5. Persist changed state or read current state.
6. Print human-readable output.

Это делает команды единообразными и не дает бизнес-логике расползаться по CLI handlers.

## Поток `help`

```text
CLI -> HelpProvider -> Output
```

Это самый легкий сценарий.

Он не должен:

- инициализировать runtime;
- читать все task files;
- поднимать executor adapters.

## Поток `template list` и `template show`

```text
CLI -> TemplateRegistry -> Output
```

Здесь CLI читает built-in templates и пользовательские шаблоны проекта, но ничего не меняет в runtime.

## Поток `task create`

```text
CLI -> TaskService.Create -> Domain validation -> TaskStore.Save -> Output
```

Основные шаги:

1. CLI получает `title`, `goal`, `agent`, `prompt_templates` и `scope`.
2. Task service создает `Task`.
3. Domain валидирует обязательные поля и совместимость шаблонов.
4. Task store пишет `.agentctl/tasks/TASK-001.yml`.
5. CLI выводит `task_id` и стартовый статус.

## Поток `task run`

```text
CLI -> TaskService.Run -> ContextPackBuilder -> PromptBuilder -> RuntimeManager -> AgentExecutor
```

Основные шаги:

1. Загружается `task.yml`.
2. Проверяется текущий статус задачи.
3. Собирается context pack.
4. Применяются built-in и custom templates.
5. Формируется execution contract.
6. Создается `Run`.
7. Runtime manager регистрирует активный run и lock.
8. Executor получает команду на выполнение.

## Рекомендуемый execution contract

Для этой CLI логичен файловый контракт, а не длинный inline prompt.

Исполнитель получает короткую инструкцию вида:

```text
Приступай к выполнению задачи: .agentctl/tasks/TASK-001.yml
Контекст: .agentctl/context/TASK-001/
Артефакты сохраняй в: .agentctl/runs/TASK-001/RUN-001/
```

Это хорошо сочетается с task-based моделью:

- источник истины остается в файлах;
- prompt становится коротким и воспроизводимым;
- `.yml`-уточнения автоматически входят в контекст;
- built-in templates влияют на поведение агента, а не на форму пользовательской команды.

## Поток встроенного шаблона `clarify_if_needed`

Этот поток не должен быть отдельной веткой `task questions`. Это поведение шаблона в рамках общего `task run`.

```text
RunTask -> Template behavior -> ClarificationRequest created -> Status = needs_clarification
```

Шаги:

1. Агент анализирует задачу и подключенный контекст.
2. Если все понятно, выполнение продолжается.
3. Если задача неоднозначна, создается `clarification_request_001.yml`.
4. Задача переводится в `needs_clarification`.
5. Активное исполнение завершается без внесения изменений.

## Поток `clarification generate`

```text
CLI -> ClarificationService.Generate -> ClarificationStore.Save -> Output path
```

Шаги:

1. Находится `pending_request`.
2. На его основе строится editable `clarification_001.yml`.
3. Файл сохраняется в `.agentctl/clarifications/TASK-001/`.
4. CLI выводит путь к созданному файлу.

## Поток `clarification attach`

```text
CLI -> ClarificationService.Attach -> Domain transition -> TaskStore.Update -> EventSink
```

Шаги:

1. Файл уточнения валидируется.
2. Он добавляется в список attached clarifications.
3. `pending_request` закрывается.
4. Статус задачи меняется на `ready_to_resume`.
5. Событие пишется в event stream.

## Поток `task resume`

```text
CLI -> TaskService.Resume -> ContextPackBuilder -> PromptBuilder -> RuntimeManager -> AgentExecutor
```

Шаги:

1. Загружается задача в статусе `ready_to_resume`, `paused` или `stopped`.
2. Context pack пересобирается с учетом attached clarifications.
3. Создается новый run или продолжается существующий, если backend это поддерживает.
4. Исполнение возвращается в `running`.

## Поток `task ps`

```text
CLI -> RuntimeQuery.ListActive -> RuntimeStore.Read -> Output
```

Команда должна читать только runtime registry и не загружать весь task archive.

Она показывает:

- task id;
- run id;
- agent;
- статус;
- heartbeat;
- активный шаг пайплайна.

## Поток `task inspect`

`inspect` должен быть read-only use case:

1. загрузить `task.yml`;
2. загрузить последний `run`;
3. загрузить runtime metadata;
4. загрузить attached clarifications;
5. вернуть агрегированную карточку.

## Поток `task watch`

```text
CLI -> RuntimeQuery.Stream -> EventStore.Tail -> Output
```

`watch` должен читать `events.ndjson` и heartbeat, а не опрашивать executor напрямую.

## Поток `task stop`

```text
CLI -> RuntimeControl.Stop -> RuntimeBackend.SignalStop -> Status transition
```

Смысл:

- graceful signal;
- завершение на безопасной границе;
- сохранение максимума артефактов;
- переход `running -> stopping -> stopped`.

## Поток `task kill`

```text
CLI -> RuntimeControl.Kill -> RuntimeBackend.SignalKill -> Forced status transition
```

Смысл:

- аварийная остановка;
- немедленное завершение;
- возможны неполные артефакты;
- переход `running -> killed`.

## Поток `task cancel`

```text
CLI -> TaskService.Cancel -> Domain validation -> TaskStore.Update
```

Эта команда нужна не для живого процесса, а для задач, которые больше не должны продолжаться.

Типовые случаи:

- `draft`;
- `queued`;
- `needs_clarification`;
- `ready_to_resume`.

## Почему потоки важно фиксировать отдельно

Если не описать command flows явно, логика CLI быстро расползется:

- часть правил уйдет в transport;
- часть в storage;
- часть в executor adapters;
- часть в случайные if-ветки разных команд.

Выделенные потоки удерживают архитектуру чистой и делают Go-код предсказуемым.
